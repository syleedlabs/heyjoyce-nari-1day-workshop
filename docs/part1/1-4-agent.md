# 1-4. 자동화 맛보기 : 미니 코치 첫 실행

::: tip Step 4 / 9 · Part 1
1-3에서 강사 에이전트를 봤으니, 이제 나리님의 **커리어 코치를 직접 한 번** 돌려봅니다. Part 2에서 제대로 만들 코치의 **미니 버전**이에요. 오늘의 첫 성공이에요.
:::

## 오늘 만들어볼 결과물, 먼저 봅니다

거창하게 설명하기 전에, **오늘 이 챕터에서 받아볼 결과물**을 먼저 보여드릴게요. 커리어 질문 하나에 미니 코치가 이렇게 답합니다.

```
[코치] 답변

이직 자체가 답이라기보단, 먼저 "지금 자리에서 채워지지 않는 게 뭔지"를
짚는 게 순서예요. 안정을 원하는지, 성장을 원하는지에 따라 답이 갈리거든요.

📎 근거: 커리어 앵커 (섹션 3) — "포기 못 하는 것이 결정의 기준"

💬 이어서 묻고 싶은 것: 지난 이직·선택에서 절대 포기 못 했던 건 뭐였나요?
```

핵심은 하나예요 — **📎 근거**가 붙어 있다는 것. 1-3에서 강사 에이전트가 출처를 명기하던 것과 똑같아요. 지금은 **빌린 지식(시드)**으로 답하지만, Part 2에서 나리님 자료를 넣으면 이 답이 **나리님 목소리**로 바뀝니다.

## [개념] 이 답이 나오는 파이프라인 + 도메인

### 파이프라인 흐름

강사의 콘텐츠 에이전트가 `리서치 → 팩트체크 → 제작`으로 흘렀듯, 미니 코치도 정해진 흐름으로 답을 만들어요.

```
커리어 질문  →  지식(시드) 검색  →  맞는 프레임 매칭  →  답변 + 📎 출처
                                                    └ (자료 밖이면 "일반론" 라벨)
```

질문을 받으면 아무 말이나 하는 게 아니라, **가진 지식을 먼저 뒤져 근거를 찾고**, 그 근거를 밝히며 답해요. 근거가 없으면 "제 자료엔 없어요"라고 정직하게. 이 흐름이 코치를 믿을 수 있게 만드는 전부예요.

### 미니 코치가 아는 것 — 커리어 코칭 도메인

이 미니 코치가 다루는 영역은 **커리어 코칭**이에요 — 이직·직무 전환·커리어 설계 같은 고민이요. 아직 나리님 자료를 넣기 전이라, 이 워크북엔 커리어 코칭에서 널리 쓰는 **공개 프레임워크 3개**가 시드(예제 지식)로 들어 있어요. 미니 코치는 이걸 근거로 답합니다.

| 프레임워크 | 무엇을 보는 도구 | 잘 맞는 질문 |
|---|---|---|
| **이키가이** | 좋아함·잘함·세상필요·보상 4가지가 겹치는 지점으로 방향 찾기 | "지금 일이 안 맞는 것 같아요" |
| **GROW 모델** | 목표→현실→선택지→실행의지 순서로 대화를 굴리기 | "뭐부터 해야 할지 막막해요" |
| **커리어 앵커** | "절대 포기 못 하는 것" 8가지로 갈림길 결정을 진단 | "이직할지 말지 모르겠어요" |

(이 자료는 `knowledge/_시드_공개프레임워크/` 폴더에 들어 있어요.)

::: warning 이건 아직 '빌린' 도메인이에요
이 프레임워크들은 **일반 공개 이론**이라, 코치가 인용할 때 "일반 프레임"으로 표시돼요 — 나리님만의 관점·화법·사례는 아직 없죠. Part 2에서 나리님의 실제 강의·상담을 넣으면, 도메인이 **나리님 본인의 것**으로 바뀝니다.
:::

## [실습 1] 대화형으로 첫 답 받기 (첫 성공)

먼저 **평소 말하듯 직접 부탁**해 봅니다. 클로드 코드 터미널에 아래처럼 입력해 보세요(질문은 본인 것으로 바꿔도 좋아요).

```
knowledge/_시드_공개프레임워크 폴더의 프레임워크를 참고해서 내 커리어 질문에 답해줘.
어느 프레임에서 나왔는지 📎 근거로 밝히고, 자료에 없는 내용은 "일반론"이라고 표시해줘.
질문: "지금 회사를 계속 다녀야 할까요?"
```

코치가 시드에서 맞는 프레임을 찾아, **📎 근거를 붙여** 답합니다. 방금 위에서 본 그 결과물이 지금 나리님 손에서 나온 거예요 — **첫 성공**이에요. 🎉

::: tip 그런데… 매번 이렇게 길게?
답은 잘 나왔지만, 이 긴 부탁을 **매번 타이핑하긴 번거롭죠.** 자주 쓰는 일이라면 더더욱요. 그래서 다음 실습이 있어요.
:::

## [실습 2] 커맨드로 한 줄에 (효율성 체험)

방금 그 긴 지침을 **미리 담아둔 슬래시 커맨드**가 있어요. 한 단어면 됩니다.

```
/1-4-seed-answer
```

코치가 질문을 물어보고(못 떠올리면 샘플을 제안해요), **실습 1과 똑같이** 시드 근거로 📎 붙여 답합니다. 긴 프롬프트 없이 한 줄로요.

이게 자동화의 핵심이에요 — **자주 하는 일의 지침을 커맨드에 묶어두면**, 다음부터는 한 단어로 재실행. 강사가 `run_crawl.sh` 한 줄로 크롤을 돌렸던 것과 같은 원리예요.

## [체크포인트] 첫 성공을 확인해요

- **📎 근거가 붙은 답을 받았나요?** (실습 1·2 어느 쪽이든) 아직 본인 자료가 아니어도, 출처를 밝히며 답하는 구조를 봤다면 이 챕터의 목적은 달성입니다.
- **대화형과 커맨드의 차이를 느꼈나요?** 긴 부탁(실습 1)이 한 줄 커맨드(실습 2)로 묶이는 것 — 이게 반복 작업을 자동화하는 방식이에요.
- **"이 자리에 내 자료가 들어가면?"이 그려지나요?** 지금 시드 자리에 나리님의 강의·상담이 들어가면, 같은 답이 **본인 목소리**가 됩니다.
