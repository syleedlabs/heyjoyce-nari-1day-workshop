## [개념] Part 3에서 하는 일 — 3모드를 손에 붙이기

Part 2에서 코치의 심장(지식 베이스)을 만들었어요. 자료를 인제스트하고, 구조화하고, 2-3에서 **코치가 내 슬라이드를 인용해 답하는 첫 순간**까지 봤습니다. 그게 큰 우와 #1이었죠.

Part 3는 그 코치를 **매일 쓰는 도구 3개**로 완성하는 파트예요.

| 모드 | 커맨드 | 하는 일 | 이 챕터 |
|---|---|---|---|
| **대리응답** | `/answer` | 코치이 질문에 나리님이라면 할 답 | **3-1 (지금)** |
| **토론** | `/spar` | 내 관점에 반론을 던져 생각을 벼림 | 3-2 |
| **리서치** | `/research` | 커리어 주제를 대신 조사, 내 프레임으로 해석 | 3-3 |

그리고 3-4에서 **근거 검수**를 붙여 코치가 스스로 근거 없는 답을 걸러내게 만들고(큰 우와 #2), 3-5에서 이 3모드를 나리님 실제 일상에 꽂습니다.

## [개념] /answer — 대리응답 모드

`/answer`는 코치이(내담자)의 커리어 질문에 **나리님이라면 할 답**을 초안으로 써주는 모드예요. 핵심은 두 가지입니다.

- **그라운딩** — 답은 나리님 코퍼스(`knowledge/`)에 근거한다. 지어내지 않는다.
- **출처 표기** — 답 아래에 **어느 슬라이드·영상에서 나왔는지** 명시한다. 코퍼스 밖이면 "일반론"으로 라벨을 붙여 분리한다.

이 두 가지가 코칭에서 왜 중요하냐면 — **커리어 조언은 틀리면 사람 인생에 영향을 주기 때문**이에요. 그래서 코치는 "그럴듯한 답"보다 "근거 있는 답, 없으면 없다고 말하는 답"을 우선합니다. 이게 v5 코치의 신뢰 골격이고, 3-4에서 검수자로 한 번 더 못 박아요.

### 세 가지 근거 상태

코치는 질문을 받으면 `knowledge/_인덱스.md`와 `knowledge/원본/`을 먼저 뒤져서, 근거가 어느 상태인지에 따라 답하는 방식을 바꿔요.

| 근거 상태 | 코치의 답변 방식 |
|---|---|
| **코퍼스에 있음** | 나리님 관점으로 답 + `📎 근거: <자료명> (슬라이드 N / 영상 MM:SS)` |
| **부분만 있음** | 근거분은 출처 표기, 나머지는 `⚠️ 여기서부턴 제 자료 밖 일반론입니다` 라벨 |
| **없음** | "제 자료엔 없는 주제예요" + (원하면) 일반론을 라벨로 분리 |

이 세 상태를 코치가 **매 답변마다 스스로 구분**하는 게 대리응답의 품질이에요. 출처가 붙은 답만 "나리님 답"이고, 라벨 붙은 부분은 "일반론"이라는 걸 코치이가 한눈에 알 수 있죠.

## [실습] 실제 코치이 질문으로 /answer 돌리기

2-3에서 이미 한 번 맛보셨으니, 이번엔 **실제로 받은 적 있는 코치이 질문**으로 돌려볼게요. 지금 떠오르는 상담 질문 하나를 골라주세요 — 예: "3년차인데 지금 이직해도 될까요?", "직무를 바꾸고 싶은데 경력이 리셋될까 봐 무서워요."

::: warning 코치이 개인정보는 익명화
실제 상담 질문을 넣으실 때, 내담자 실명·회사·특정할 수 있는 정보는 빼거나 일반화해주세요. 개인정보 원칙은 [2-4](/part2/2-4-privacy)에서 다뤘고, 부록에도 상세 가이드가 있어요.
:::

클로드 코드 터미널에서 다음 슬래시 커맨드를 입력하세요.

**[커맨드 입력]**

```
/answer 3년차 마케터인데 지금 이직해도 될지 고민이에요
```

코치가 다음과 같이 답합니다.

```
[코치] 답변

<결론 — 나리님 화법으로>
지금이 "이직 타이밍"인지보다, 이직이 어떤 가설을 검증하려는 건지가 먼저예요.

<근거·설명>
3년차는 커리어에서 첫 번째 방향 재점검 구간이에요. 도망치듯 옮기면
같은 문제가 다음 회사에서 반복되기 쉽고, 가설(이 일이 나랑 맞는지)을
검증하려는 이동이면 리셋이 아니라 축적이 됩니다.
📎 근거: 「직무전환 3단계」 강의 (슬라이드 12~14)

💬 이어서 묻고 싶은 것:
지금 회사에서 "이건 검증됐다" 싶은 것과 "아직 모르겠다" 싶은 걸
하나씩 꼽아보실래요?
```

`📎 근거`가 붙어 있으면 그 답은 나리님 자료에서 나온 거예요. 나리님 슬라이드를 열어 실제로 그 내용이 있는지 대조해보세요 — **"내가 말한 거랑 똑같네"** 하는 순간이 코치가 제대로 그라운딩된 증거입니다.

### 코퍼스 밖 질문도 한 번

일부러 나리님 자료에 없을 법한 질문을 던져보세요 — 예: "미국 H-1B 비자 절차가 어떻게 되나요?"

```
/answer 미국 취업 비자 절차 알려주세요
```

코치가 이렇게 선을 긋습니다.

```
[코치] 답변

제 자료엔 없는 주제예요. 비자·이민 절차는 나리님 코칭 코퍼스
밖이라, 제가 근거를 갖고 답할 수 있는 영역이 아니에요.

⚠️ 여기서부턴 제 자료 밖 일반론입니다:
H-1B는 매년 추첨 방식이고 고용주 스폰서가 필요해요. 다만 이건
나리님 관점이 아니라 일반 정보라, 이민 전문가 확인을 권해요.
```

이 "선 긋기"가 대리응답의 핵심이에요. 코치는 **모르는 걸 아는 척하지 않고**, 넘어갈 땐 라벨을 붙여 코치이가 오해하지 않게 합니다.

## [실습] 출처 표기·라벨 다듬기 — /3-1-tune-answer

첫 실행에서 톤이나 출처 표기가 나리님 스타일과 조금 어긋날 수 있어요. 튜닝 가이드를 불러옵니다.

클로드 코드 터미널에서 다음 슬래시 커맨드를 입력하세요.

**[커맨드 입력]**

```
/3-1-tune-answer
```

코치가 자주 나오는 세 가지 튜닝을 함께 봐드립니다.

| 증상 | 다듬는 곳 |
|---|---|
| 출처 표기가 뭉뚱그려짐 (자료명만, 위치 없음) | `answer.md` §2 + 페르소나 §4 — 슬라이드 번호·영상 타임스탬프까지 |
| 코퍼스 밖인데 라벨 없이 단정함 | 페르소나 §4 그라운딩 규칙 강화 |
| 결론이 나리님 말투가 아님 | 페르소나 §3 톤·화법 다듬기 |

튜닝은 파일을 직접 고치지 않으셔도 돼요. 코치에게 한국어로 말씀하시면 됩니다 — 예: "근거 표기할 때 슬라이드 번호까지 꼭 붙여주세요." 그러면 코치가 `answer.md` 또는 페르소나를 알아서 다듬어요.

::: info 운영 커맨드 vs 실습 커맨드
- `/answer` (단어형) = 워크숍 끝나고 **평생 쓰는 운영 커맨드**
- `/3-1-tune-answer` (번호형) = 오늘 튜닝을 돕는 **실습 진행 커맨드**

워크숍 후엔 `/answer`만 기억하시면 돼요. 번호형은 오늘의 안내판입니다.
:::

---

::: tip 대리응답 완성
`/answer`가 나리님 손에 들어왔어요.

- 실제 코치이 질문으로 첫 실행 — `📎 근거` 표기 확인
- 코퍼스 밖 질문으로 "선 긋기"(일반론 라벨) 확인
- 출처 표기·톤을 나리님 스타일로 튜닝

이제 코치는 **모르는 걸 아는 척하지 않고**, 아는 건 출처를 붙여 답합니다. 다음 챕터에서는 코치가 나리님에게 **반론을 던지는** 토론 파트너로 변신해요.
:::
