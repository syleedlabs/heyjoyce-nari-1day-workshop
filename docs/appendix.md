# 부록

워크숍이 끝난 뒤에 펼쳐보는 자가 학습 자료예요. 자료를 더 넣으실 때, 개인정보를 정리하실 때, 채널을 확장하실 때, 코치가 이상하게 굴 때 — 상황에 맞는 섹션만 골라 보시면 됩니다.

| 섹션 | 언제 펼쳐보세요 |
|---|---|
| **A. 자료 인제스트 레시피** | 강의 PPT·영상·상담 노트를 코치에 더 넣을 때 |
| **B. 개인정보 익명화 가이드** | 상담 노트를 넣기 전, 공개 레포에 뭐가 올라가는지 확인할 때 |
| **C. MCP 옵션 확장** | Gmail·Drive·Notion을 코치에 연결할 때 |
| **D. 트러블슈팅** | 코치가 근거 없이 지어내거나 자료를 못 찾을 때 |
| **E. 워크숍 후 다음 단계** | 코치를 계속 키우고 싶을 때 |
| **F. 에이전트를 이루는 .md 파일들** | 코치가 어떤 파일들로 움직이는지, 뭘 고칠 때 어딜 여는지 알고 싶을 때 |
| **G. 참고·출처** | 이 워크북·코치가 쓴 프레임워크·데모의 출처를 확인할 때 |

## A. 자료 인제스트 레시피

코치의 지식은 `knowledge/원본/`에 넣은 자료만큼이에요. 워크숍에선 대표 3~5개로 시작했지만, 자료를 더 넣을수록 코치는 나리님을 더 닮아갑니다. 자료 종류별로 텍스트로 바꾸는 법을 정리했어요.

### A-1. PPT → 텍스트

강의 슬라이드는 텍스트로 바꿔서 `knowledge/원본/`에 넣어요. 몇 가지 방법이 있습니다.

| 방법 | 절차 | 적합 |
|---|---|---|
| **파워포인트 내보내기** | 파일 → 내보내기 → **개요/RTF** 또는 "다른 이름으로 저장 → 개요(.rtf)" | 텍스트 위주 슬라이드 |
| **PDF로 저장 후 넣기** | PPT를 PDF로 저장 → 클로드 코드에 PDF 경로를 주면 코치가 읽음 | 도표·이미지 섞인 슬라이드 |
| **직접 붙여넣기** | 슬라이드 노트·핵심 문장을 손으로 마크다운에 정리 | 슬라이드가 몇 장 안 될 때 |

::: tip 슬라이드 번호를 함께 남기세요
코치가 `📎 근거: 「강의명」 슬라이드 12`처럼 출처를 표기하려면, 원본에 **슬라이드 번호**가 남아 있어야 정확해요. 텍스트로 옮길 때 `## 슬라이드 12` 같은 헤딩으로 구간을 표시해두면 코치의 출처 표기가 훨씬 정밀해집니다.
:::

### A-2. 영상 → 자막·전사

영상은 자막(transcript)을 추출해 텍스트로 넣어요. 영상 파일 자체를 넣는 게 아니라 **말한 내용의 텍스트**가 코치의 재료예요.

| 소스 | 자막 추출 방법 |
|---|---|
| **유튜브 (내 채널)** | 영상 → 더보기 → **스크립트 표시** → 텍스트 복사 |
| **로컬 영상 파일** | 자동 자막 도구(Whisper 등)로 전사 → 텍스트 저장 |
| **줌·웨비나 녹화** | 녹화 시 자막 켜두면 `.vtt`·`.srt` 파일 생성 → 텍스트로 변환 |

전사 텍스트엔 타임스탬프(`MM:SS`)가 남으면 좋아요. 코치가 `📎 근거: 「영상명」 12:30`처럼 구간을 짚어줄 수 있거든요.

### A-3. 상담 노트 정리

상담 경험은 코치의 가장 값진 재료지만, **개인정보가 가장 많이 든 자료**이기도 해요. 넣기 전에 반드시 익명화(B 섹션)를 거치세요. 정리 팁이에요.

- 내담자별 원본이 아니라, **패턴·프레임워크 단위**로 재구성하면 익명화도 쉽고 코치가 쓰기도 좋아요. 예: "3년차 이직 고민 케이스에서 자주 쓰는 질문 5개"
- 실명·회사명은 빼고 **상황의 구조**만 남기세요. 예: "IT 대기업 3년차 마케터 → 스타트업 고민"
- `knowledge/_인덱스.md`에 프레임워크로 올려두면 코치가 스파링·리서치 때도 참조합니다.

## B. 개인정보 익명화 가이드

코칭 자료엔 내담자 실명·회사·민감정보가 들어 있을 수 있어요. v5 코치는 **로컬 보관 + 익명화** 두 원칙으로 이걸 지킵니다. 2-4에서 다룬 내용의 상세 레퍼런스예요.

### B-1. 실명 → 가명 치환 규칙

인제스트 전에 다음 항목을 치환하세요.

| 원본 | 치환 규칙 | 예시 |
|---|---|---|
| 내담자 실명 | 가명 또는 역할로 | "김OO" → "A씨" 또는 "3년차 마케터" |
| 회사명 | 업종·규모로 일반화 | "네이버" → "IT 대기업" |
| 특정할 수 있는 디테일 | 구조만 남기고 제거 | "작년 대상 받은 그 프로젝트" → "성과를 낸 프로젝트" |
| 연락처·이메일·주소 | 전량 삭제 | — |

핵심 원칙: **누가 봐도 특정 개인을 식별할 수 없어야** 해요. 이름만 지웠는데 "그 회사 그 직책"으로 특정되면 익명화가 안 된 거예요. 상황의 **구조**만 남기고 개인 식별 정보는 지웁니다.

### B-2. .gitignore 확인 — 뭐가 올라가고 뭐가 안 올라가나

`knowledge/` 실자료는 **절대 공개 레포에 올리지 않아요.** `.gitignore`에 이미 등록돼 있는지 확인하세요.

::: details .gitignore 확인 절차
프로젝트 루트의 `.gitignore`에 다음 줄이 있어야 해요.

```
knowledge/원본/
knowledge/_인덱스.md
HABITS.md
```

확인이 어려우시면 코치에게 물어보세요 — "knowledge 폴더가 git에 안 올라가는지 확인해줘." 코치가 `.gitignore`를 점검하고, 빠져 있으면 추가해드립니다.
:::

**공개 레포에 올라가는 것 / 안 올라가는 것**을 분명히 해둘게요.

| 올라감 (공개 OK) | 안 올라감 (로컬 전용) |
|---|---|
| 워크북·템플릿 구조 (`docs/`) | `knowledge/원본/` (실제 강의·상담 자료) |
| 페르소나 **틀** (`.claude/agents/코치.md`) | `knowledge/_인덱스.md` (구조화된 지식) |
| 운영 커맨드 (`.claude/commands/`) | `HABITS.md` (개인 워크플로) |

::: warning 페르소나 파일에도 민감정보 금지
페르소나(`코치.md`)는 공개될 수 있어요. 여기에 실제 내담자 정보나 민감한 상담 내용을 적으면 안 됩니다. 페르소나엔 나리님의 **관점·프레임워크·톤**만 담고, 구체적 상담 내용은 로컬 `knowledge/`에만 두세요.
:::

## C. MCP 옵션 확장

v5 코치의 코어는 **로컬 파일**이라 실계정 연결 없이도 완결돼요. 아래는 필요할 때 얹는 **선택** 확장이에요. 각 30분 안에 붙습니다.

### C-1. Gmail — 코치이 질문 수신·답변 초안 {#gmail-recipe}

**용도**: 코치이가 메일로 보낸 커리어 질문을 코치가 읽고 `/answer` 초안을 임시보관함에 저장.

::: details 절차
1. Gmail MCP 추가 → 본인 계정으로 OAuth 동의 (읽기 + 쓰기)
2. `.claude/commands/`에 `inbox-answer.md` 생성:
   ```markdown
   ---
   description: Gmail의 코치이 질문 메일을 읽어 /answer 초안을 임시보관함에 저장
   ---
   - mcp__gmail로 미답 커리어 질문 메일 조회
   - 각 질문에 /answer 로직 적용 (그라운딩 + 출처 표기)
   - 답변 초안을 임시보관함 저장 (자동 발송 X)
   ```
3. **자동 발송 금지** — 코치는 초안까지만, 발송은 나리님이 직접 검토 후. 페르소나 §7 안전 정책과 동일해요.
:::

### C-2. Google Drive — 자료 소스 연결 {#drive-recipe}

**용도**: 강의 PPT·영상 자막이 Drive에 있으면, 매번 내려받지 않고 코치가 직접 읽어 인제스트.

::: details 절차
1. Google Drive MCP 추가 → OAuth 동의 (읽기)
2. 코치에게: "내 Drive의 '강의자료' 폴더를 knowledge에 인제스트해줘"
3. 코치가 Drive 파일을 읽어 `knowledge/원본/`에 텍스트로 정리 (익명화 규칙 B 적용)
:::

### C-3. Notion — 자료 소스 연결 {#notion-recipe}

**용도**: 코칭 프레임워크·상담 메모를 Notion에 정리해두셨다면, 그 페이지를 코치가 읽어 `knowledge/_인덱스.md`에 반영.

::: details 절차
1. Notion MCP 추가 → 워크스페이스로 OAuth 동의 (읽기)
2. 코치에게: "내 Notion '코칭 프레임워크' 페이지를 지식 베이스에 반영해줘"
3. 코치가 페이지를 읽어 구조화 → `_인덱스.md` 업데이트 (개인정보는 B 규칙으로 익명화)
:::

## D. 트러블슈팅

### D-1. 코치가 근거 없이 지어낼 때

가장 조심해야 할 문제예요. 코치가 출처 없는 숫자나 자료에 없는 주장을 나리님 답인 것처럼 낼 때.

| 증상 | 원인 | 해결 |
|---|---|---|
| 출처 없는 수치를 단정 | 페르소나 §4 그라운딩 규칙이 느슨함 | 페르소나 §4 강화 — "출처 없는 수치는 '확인 안 됨'으로" 명시 |
| 코퍼스 밖인데 라벨 없이 답 | 코퍼스 밖 처리 규칙이 약함 | 페르소나 §4에 "밖이면 '일반론' 라벨 필수" 강조 |
| 검수를 통과했는데도 샘 | 검수자가 너무 후함 | [2-4](/part2/2-4-build) 그라운딩 검수자 재점검 — `그라운딩-검수.md`를 "의심 우선"으로 |

근본 처방은 **페르소나 §4(그라운딩 규칙)를 계속 조이는 것**이에요. 코치가 지어내는 순간을 목격할 때마다 그 케이스를 페르소나에 규칙으로 추가하면, 같은 실수가 줄어듭니다.

### D-2. 코치가 자료를 못 찾을 때

나리님 자료에 분명 있는데 코치가 "제 자료엔 없어요"라고 할 때.

| 증상 | 원인 | 해결 |
|---|---|---|
| 있는 내용을 "없다"고 함 | `knowledge/_인덱스.md`에 그 주제가 안 잡혀 있음 | 2-4 구조화로 돌아가 인덱스 보강 — 그 프레임워크를 인덱스에 추가 |
| 자료는 찾는데 엉뚱한 구간 인용 | 원본에 구간 표시(슬라이드/타임스탬프)가 없음 | A-1·A-2 방식으로 구간 라벨을 원본에 추가 |
| 자료가 아예 검색 안 됨 | `knowledge/원본/`에 파일이 없거나 빈 파일 | 인제스트가 됐는지 확인, A 레시피로 재인제스트 |

`_인덱스.md`가 코치의 지도예요. 코치가 자꾸 못 찾으면, 대개 인덱스가 부실한 거예요. **인덱스를 보강하는 게 첫 처방**입니다.

## E. 워크숍 후 다음 단계

오늘 만든 코치를 며칠·몇 주에 걸쳐 키우는 로드맵이에요.

- **전체 코퍼스 인제스트 (숙제)** — 오늘 못 넣은 강의·영상·상담 노트를 A 레시피로 마저 넣기. 자료가 쌓일수록 코치의 그라운딩이 두터워져요.
- **습관 굳히기** — [2-5](/part2/2-5-tips)에서 만든 `HABITS.md` 트리거 중 **하나**를 이번 주에 시작. 하나가 손에 붙은 다음 다음 걸 추가.
- **페르소나 진화** — 코치 답이 나리님 직관과 어긋날 때마다 한 줄로 교정. 한 달 다듬으면 나리님에게만 맞는 코치가 됩니다.
- **채널 확장 (선택)** — C 섹션의 Gmail·Drive·Notion을 필요할 때 하나씩.

::: tip 오늘의 코치는 시작점이에요
워크숍에서 만든 건 완성품이 아니라 **자라나는 코치의 씨앗**이에요. 자료를 넣을수록 나리님을 닮아가고, 쓸수록 나리님 일상에 녹아듭니다. 이번 주에 딱 한 번, 상담 30분 전에 `/research`를 불러보는 것부터 시작하세요.
:::

## F. 에이전트를 이루는 .md 파일들 {#md-hierarchy}

코치는 하나의 파일이 아니라 **여러 마크다운 파일의 협업**으로 움직여요. 각 파일이 무슨 역할이고 언제 읽히는지 알면, 코치를 고칠 때 "어디를 고쳐야 하지?"가 명확해집니다.

### F-1. 네 가지 파일과 역할

| 파일 | 범위 | 역할 | 언제 읽히나 |
|---|---|---|---|
| `CLAUDE.md` | 프로젝트 전체 | 늘 지켜야 할 규칙·맥락 (가장 큰 틀) | 항상 |
| `.claude/agents/코치.md` | 이 코치 | **페르소나** — 정체성·화법·그라운딩 규칙 | 코치가 일할 때 |
| `.claude/commands/*.md` | 이 커맨드 | **슬래시 커맨드** 실행 지침(`/answer`·`/spar`·`/research`) | 그 커맨드를 부를 때 |
| `SKILL.md` | 특정 작업 | **스킬** — 재사용 가능한 작업 절차 패키지 | 필요할 때만 로드 |

### F-2. 큰 틀 → 작은 틀 (하이어라키)

```
CLAUDE.md                    ← 프로젝트 상시 규칙 (항상 적용, 가장 넓음)
  └ .claude/agents/          ← 페르소나 (이 코치는 누구인가)
       └ .claude/commands/   ← 커맨드 (지금 이 순간 무슨 일을 하나)
            └ SKILL.md        ← 스킬 (그 일에 필요한 특수 절차, 있을 때만)
```

큰 틀이 작은 틀을 감싸요. `CLAUDE.md`의 규칙이 늘 배경에 깔리고, 그 위에서 **페르소나**가 "코치는 누구인가"를 정하고, **커맨드**가 "지금 무슨 일을 하는가"를 정하고, **스킬**은 그 일에 특수 절차가 필요할 때만 불려옵니다.

### F-3. 그래서 고칠 때 — 어디를 여나

| 바꾸고 싶은 것 | 고치는 파일 |
|---|---|
| 코치 말투·관점·철칙 | `.claude/agents/코치.md` (페르소나) |
| 답변 형식·출처 표기 방식 | 해당 `.claude/commands/*.md` (커맨드) |
| 프로젝트 전체에 늘 적용할 규칙 | `CLAUDE.md` |
| 코치의 지식 내용 | `knowledge/` (파일 형태의 지식 자체) |

::: tip SKILL.md·CLAUDE.md는 지금 없어도 괜찮아요
오늘 만드는 코치는 **페르소나 + 커맨드**로 충분해요.
- `SKILL.md`(스킬)는 이 워크숍에선 안 씁니다 — 반복하는 복잡한 작업을 패키지로 묶는 확장 방식인데, 나중에 코치가 커지면 유용해져요.
- `CLAUDE.md`도 지금 배포본엔 **아직 없을 수 있어요.** 프로젝트 전체에 늘 적용할 규칙이 생기면 그때 루트에 만들면 됩니다 — 코치에게 "CLAUDE.md 만들어줘" 하면 잡아줘요.
:::
