# 2-3. 코치의 첫 그라운딩 답변

::: tip Step 7 / 13 · Part 2
이 챕터에서 — **코치가 본인 자료를 인용해(📎 근거 표기) 실제 커리어 질문에 처음으로 답합니다. "내가 말한 거랑 똑같네" 하는 순간이에요.**
:::

## 프롤로그의 약속이 실현되는 순간

워크숍을 열며 이런 약속을 드렸어요 — *"4시간 후, 당신의 코치가 당신 자료로 이렇게 답합니다."* 프롤로그에서 본 건 **강사의** 코치가 **강사의** 슬라이드를 인용해 답하는 모습이었죠.

이 챕터에서 그 약속이 **본인 것으로** 실현됩니다. 지금까지 코치에게 정체성(Part 1)과 지식(2-1·2-2)을 줬어요. 이제 그 코치를 실제 커리어 질문 앞에 세우고, 본인 자료를 인용해 답하게 합니다. 코치가 내놓는 답을 읽으며 *"어, 이거 내가 상담에서 하던 말이랑 똑같네"* 하고 느끼는 순간 — 이게 이 워크숍 전체에서 첫 번째 큰 우와예요. 코치가 본인의 **분신**임을 처음으로 실감하는 지점입니다.

## 그라운딩 = 출처를 반드시 밝힌다

이 답변이 프롤로그 데모와 다른 결정적 한 가지는 **출처 표기**예요.

코치는 답할 때마다 어느 자료의 어느 부분에서 그 말이 나왔는지를 밝힙니다. 이걸 **그라운딩(grounding)** 이라고 불러요 — 답을 본인 코퍼스에 "접지"시킨다는 뜻입니다.

| 근거 상태 | 코치의 행동 |
|---|---|
| **자료에 근거 있음** | 답변 + `📎 근거: <자료명> (슬라이드 N / 영상 MM:SS)` |
| **자료에 일부만 있음** | 근거 있는 부분은 출처 표기 + 나머지는 `⚠️ 여기서부턴 제 자료 밖 일반론입니다` 라벨 |
| **자료에 없음** | `제 자료엔 없는 주제예요` 라고 솔직히 + (원하면) 일반론임을 라벨로 붙여서만 |

왜 이렇게까지 엄격하냐면, 커리어 조언은 **틀리면 사람 인생에 영향**을 주기 때문이에요. 근거 없이 그럴듯하게 지어낸 답(할루시네이션)은 코칭에서 가장 위험합니다. 출처 표기는 그걸 막는 **1차 방어선**이에요. 코치가 지어냈는지 본인 자료에 근거했는지가 답만 봐도 한눈에 보이니까요. (근거 없는 답을 코치가 스스로 걸러내는 2차 방어선 — 검수 — 은 [Part 3](/part3/3-4-grounding-check)에서 붙입니다.)

::: tip 출처 표기가 곧 신뢰
"📎 근거: 이직_가설검증_강의 (슬라이드 7)"이 붙은 답과, 아무 근거 없는 답. 코치이(내담자)에게 전할 때 어느 쪽을 믿고 쓰시겠어요? 출처 표기는 코치를 **믿고 쓸 수 있게** 만드는 장치예요.
:::

## [실습 1] 첫 그라운딩 답변 받기

이제 코치를 실제 질문 앞에 세웁니다. 본인이 상담에서 자주 받는 커리어 질문 하나를 떠올려주세요. (예: "지금 회사를 더 다녀야 할까요?", "직무를 바꾸는 게 맞을까요?") 떠오르는 게 없으면 커맨드가 샘플 질문을 제안합니다.

클로드 코드 터미널에서 다음 슬래시 커맨드를 입력하세요.

**[커맨드 입력]**

```
/2-3-first-grounded-answer
```

코치가 `knowledge/_인덱스.md`와 `원본/` 자료를 검색해, 본인 관점으로 답하고 출처를 붙입니다. 출력은 이런 형태예요.

```
[코치] 답변

<결론 — 본인 화법으로>

<근거·설명>
📎 근거: <자료명> (슬라이드 N / 영상 MM:SS)

(자료 밖일 때) ⚠️ 여기서부턴 제 자료 밖 일반론입니다: <내용>

💬 이어서 묻고 싶은 것: <다음 질문 1개>
```

답을 받으면 **세 가지**를 천천히 확인해주세요.

- **말투가 본인 같나요?** 코치의 어조·표현이 본인이 상담에서 쓰는 말투와 닮았는지. 이게 "분신" 여부를 가르는 감각이에요.
- **📎 근거가 붙어 있나요?** 그리고 그 근거가 **진짜 본인 자료**를 가리키나요? 실제로 그 슬라이드에서 한 말이 맞는지 대조해보세요.
- **자료 밖으로 넘어갈 때 라벨을 붙였나요?** 코퍼스에 없는 내용을 마치 본인 지식인 양 단정하지 않고 "일반론"으로 구분했는지.

세 가지가 대부분 맞으면 코치의 분신다움이 자리 잡은 거예요. 어긋나는 부분은 다음 톤 튜닝에서 다듬습니다.

## [실습 2] 톤 튜닝 — 결과 보고 코치 다듬기

첫 답을 읽으면 "이 부분은 이렇게 바뀌면 좋겠다"가 보일 거예요. 코치는 **대화로** 수정됩니다 — 파일을 직접 열 필요 없어요.

무엇을 고치느냐에 따라 수정되는 파일이 달라져요.

| 수정 요청 종류 | 어디가 수정되나 |
|---|---|
| **말투·화법·고유 관점** | 페르소나 파일 (`.claude/agents/코치.md`) |
| **출력 형식·출처 표기 방식·다음 질문 유무** | 운영 커맨드 (`.claude/commands/answer.md`) |
| **자료 내용이 틀렸거나 빠짐** | 지식 베이스 (`knowledge/원본/...` 또는 `_인덱스.md`) |

::: tip 자주 나오는 튜닝 요청 예시
- "말투가 너무 딱딱해요. 더 따뜻하게, 공감 한 문장 먼저." → 페르소나 §3 수정
- "📎 근거에 슬라이드 번호까지 꼭 넣어주세요." → 운영 커맨드 출력 형식 수정
- "이 답, 제 자료엔 없는 내용인데 근거가 붙었어요." → 지식 베이스 대조 후 코치가 근거를 뗌 (이게 그라운딩이 작동하는 증거예요)
- "'이어서 묻고 싶은 것'은 빼주세요." → 운영 커맨드 출력 형식 수정
:::

수정 요청을 대화로 전하면 코치가 해당 파일을 자동으로 고칩니다.

## [실습 3] 다시 실행 — 달라진 답 비교

튜닝이 끝나면 **같은 질문으로 한 번 더** 답을 받아보세요.

**[커맨드 입력]**

```
/2-3-first-grounded-answer
```

같은 질문인데 말투·출처 표기·형식이 달라진 게 보일 거예요. 이 **결과 받기 → 코치 다듬기 → 다시 실행**의 짧은 사이클이 코치를 본인답게 길들이는 가장 빠른 방법입니다. 오늘 이 한 번을 돌려봤으니, 앞으로 3모드를 완성할 때(Part 3) 같은 리듬으로 다듬으면 돼요.

## [체크포인트] 분신을 처음 만난 순간

- **본인 자료를 인용한 답을 받았나요?** 📎 근거가 실제 본인 자료를 가리키는 답 하나를 받았다면, 이 챕터의 핵심은 달성입니다.
- **"내가 말한 거랑 비슷하네"를 느꼈나요?** 완벽히 똑같지 않아도 괜찮아요. 그 방향으로 가고 있다는 감각이면 충분합니다. 나머지는 튜닝으로 좁혀집니다.
- **한 사이클을 돌려봤나요?** 첫 답 → 튜닝 → 다시 실행을 한 번 경험했다면, 코치를 평생 다듬어갈 방법을 손에 넣은 거예요.

:::
