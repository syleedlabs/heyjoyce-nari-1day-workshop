# 2-4. 개인정보 익명화 · 로컬 보관

::: tip Step 8 / 13 · Part 2
이 챕터에서 — **상담 노트의 실명·회사를 익명화하고, `knowledge/원본/`을 공개에서 제외해(`.gitignore`) 코치 지식을 본인 컴퓨터에만 안전하게 둡니다.**
:::

## 코칭 데이터는 특별히 민감합니다

지금까지 코치의 지식 베이스를 신나게 채웠어요. 그런데 그 안에는 다른 자료와 성격이 다른 게 하나 섞여 있습니다 — **상담 노트**예요.

상담 노트에는 실제 내담자의 이야기가 담겨 있어요. 이름, 다니는 회사, 연봉, 이직 사정, 때로는 매우 사적인 고민까지. 이건 본인의 지식 자산인 동시에, **타인의 개인정보**입니다. 그래서 이 챕터에서는 잠깐 멈춰서 두 겹의 안전장치를 겁니다 — **익명화**와 **로컬 보관**.

::: tip 무섭게 들리지만, 사실 간단해요
"개인정보"라는 말이 부담스러울 수 있어요. 하지만 여기서 하는 건 딱 두 가지뿐입니다 — (1) 노트 속 실명·회사를 가명으로 바꾸고, (2) 그 파일들이 인터넷에 올라가지 않도록 잠가두는 것. 코치가 스캔부터 처리까지 도와주니, 본인은 확인만 하면 됩니다.
:::

## 안전장치 1 — 익명화

익명화는 노트 속 **누구인지 특정할 수 있는 정보**를 지우거나 일반화하는 일이에요. 코칭 내용(프레임워크·조언·패턴)은 그대로 두고, 사람만 가립니다.

| 지울/바꿀 것 | 이렇게 |
|---|---|
| 내담자 실명 | `A님`, `내담자 B` 같은 익명 표기 |
| 회사명 | `대기업 C`, `초기 스타트업` 같은 일반화 |
| 전화·이메일·주소 | 삭제 |
| 특정 프로젝트·상품명 등 식별 단서 | 일반적 표현으로 |

핵심 원칙은 **"이 노트가 유출돼도 누구 이야기인지 알 수 없게"** 예요. 코칭의 교훈은 남기되, 사람은 안 보이게 하는 거죠. 2-1에서 노트를 넣을 때 눈에 띄는 것만 가볍게 익명화했다면, 지금은 빠진 데가 없는지 코치가 꼼꼼히 훑습니다.

## 안전장치 2 — 로컬 보관 (`.gitignore`)

두 번째 장치는 **애초에 인터넷에 안 올라가게** 막는 거예요.

워크숍이 끝나면 워크북과 코치 구조(페르소나·커맨드)는 GitHub에 공개 배포됩니다. 다른 사람도 이 워크숍을 볼 수 있게요. 하지만 **본인의 실제 지식 자료(`knowledge/원본/`)는 절대 함께 올라가면 안 돼요.**

그래서 `.gitignore`라는 파일에 `knowledge/원본/`을 등록해 뒀습니다. 이 목록에 적힌 폴더는 Git이 **아예 없는 것처럼 취급**해서, 공개 레포에 올릴 때 통째로 빠져요. 결과적으로:

- **공개되는 것** — 워크북, 코치 페르소나 틀, 커맨드, 빈 지식 지도 구조
- **본인 컴퓨터에만 남는 것** — `원본/`의 실제 강의·영상·상담 자료

::: warning `_인덱스.md`는 어떻게 하나요?
구조화한 지식 지도(`_인덱스.md`)는 기본적으로 구조라서 공개돼도 무방하지만, 본인이 채운 실제 프레임워크·상담 사례가 민감하다고 느껴지면 이 파일도 제외할 수 있어요. 실습에서 코치가 물어봅니다. 조금이라도 망설여지면 **제외하는 쪽**을 권합니다.
:::

## [실습] 개인정보 잔재 점검 + 잠금 확인

코치가 `원본/` 폴더를 훑어 익명화가 빠진 곳을 찾아주고, `.gitignore` 잠금이 제대로 걸렸는지 확인해줍니다.

클로드 코드 터미널에서 다음 슬래시 커맨드를 입력하세요.

**[커맨드 입력]**

```
/2-4-anonymize-check
```

코치가 다음을 수행합니다.

1. `knowledge/원본/`의 모든 파일을 스캔해 **실명·회사명·전화·이메일 등 개인정보 잔재**를 찾아 목록으로 보여줌
2. 발견된 항목을 어떻게 익명화할지 제안 → 본인이 확인하면 파일 수정
3. `.gitignore`에 `knowledge/원본/`이 등록돼 있는지 확인 (안 돼 있으면 추가 제안)
4. 필요하면 `_인덱스.md`도 제외할지 물어봄

::: tip 코치가 못 잡는 것도 있어요
자동 스캔은 이름·회사·연락처 같은 명백한 패턴을 잡습니다. 하지만 "그 유명한 이직 사례 있잖아요, 작년에 화제됐던…" 처럼 **맥락으로만 특정되는 정보**는 코치가 놓칠 수 있어요. 스캔 결과를 본인 눈으로 한 번 더 훑어, 사람이 떠오르는 대목이 없는지 확인해주세요.
:::

## [체크포인트] 안심하고 다음으로

- **개인정보 잔재 목록이 비었나요?** 코치 스캔 결과가 "잔재 없음"으로 나오거나, 발견된 항목을 모두 익명화 처리했다면 1차 통과입니다.
- **본인 눈으로도 훑었나요?** 맥락으로 특정되는 정보가 없는지 한 번 더 확인.
- **`.gitignore`에 `knowledge/원본/`이 있나요?** 이게 있어야 배포 때 실자료가 빠집니다. 코치가 확인해준 결과를 봐주세요.

이 세 가지가 되면, 코치의 지식 베이스는 **본인 컴퓨터 안에서만 사는** 안전한 상태가 됩니다. 강력한 분신을 만들면서도 타인의 정보와 본인의 자산을 지키는 방법을 갖춘 거예요.

- `knowledge/원본/` 개인정보 잔재 점검·익명화
- `.gitignore`로 실자료 공개 제외 확인
