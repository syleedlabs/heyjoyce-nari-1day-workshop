# 2-1. 에이전트

::: tip Step 5 / 9 · Part 2
이 챕터에서 — **에이전트·서브에이전트·오케스트레이션이 각각 무엇인지**, 그리고 우리 커리어 코치가 이 그림 안에서 어디에 서 있는지 함께 정리해봅시다.
:::

::: tip 방금 본 것들이 다 '에이전트'예요
1-3에서 본 강사의 딜 아카이브 — **리서치하는 부분, 제안하는 부분, 콘텐츠 만드는 부분** — 그리고 1-4에서 돌린 **미니 코치**. 이게 전부 이 챕터에서 말하는 "에이전트"예요. 새 개념을 배우는 게 아니라, **이미 본 것들에 이름표**를 다는 시간이에요.
:::

## [개념] 팀으로 비유하면 한 번에 잡혀요

에이전트라는 말이 처음엔 추상적으로 들릴 수 있어요. 그래서 **팀**으로 비유해봅니다. 나리님이 작은 팀을 운영한다고 생각하면, 세 가지 개념이 아래처럼 딱 나뉩니다.

| 개념 | 팀 비유 | 한 줄 정의 |
|---|---|---|
| **에이전트** | 독립적 역할·업무를 가진 **정규 팀원** | 자기 프롬프트와 도구를 갖고 스스로 판단해 일하는 1급 실행 단위 |
| **서브에이전트** | 필요할 때 부르는 **임시 지원 인력** | 특정 작업만 잠깐 맡고, 끝나면 사라지는 보조 일꾼 |
| **오케스트레이션** | 팀원을 **배치·조율하는 팀장** | 여러 에이전트를 어떻게 나눠 붙일지 정하는 조율 방식 |

정규 팀원(에이전트)은 이름·역할·책임이 있고 계속 그 자리에 있어요. 임시 인력(서브에이전트)은 "이 검토만 부탁해요" 하고 불렀다가 일이 끝나면 돌려보냅니다. 팀장(오케스트레이션)은 누구에게 무엇을 맡길지 그림을 그려요.

이 세 가지를 하나씩 풀어볼게요.

## [개념] 에이전트 — 독립적 역할을 가진 정규 팀원

에이전트는 **AI + 도구 + 판단력**의 조합입니다. 세 가지가 다 있어야 에이전트예요.

- **AI** — 언어를 이해하고 답을 생성하는 두뇌.
- **도구** — 파일을 읽고, 검색하고, 커맨드를 실행하는 손발.
- **판단력** — "지금은 자료를 먼저 찾고, 그다음 답하자" 같은 **스스로의 결정**.

그냥 질문에 답만 하는 챗봇과 다른 지점이 바로 **판단력**이에요. 에이전트는 자기 프롬프트(역할 정의)를 갖고, 상황에 맞게 도구를 골라 씁니다.

::: tip 우리 코치가 바로 에이전트예요
나리님의 커리어 코치는 하나의 정규 팀원입니다. `.claude/agents/코치.md`에 역할과 철칙이 적혀 있고, `knowledge/` 폴더를 도구로 뒤져 근거를 찾고, "이건 자료에 있으니 답하고, 저건 자료 밖이니 라벨을 붙이자"를 스스로 판단해요.
:::

### 한 팀원이 쓰는 세 가지 일하는 방식

여기서 v5의 특징이 하나 있어요. 우리 코치는 **에이전트 하나(한 페르소나)** 인데, 상황에 따라 **세 가지 방식**으로 일합니다. 팀원 여러 명이 아니라, **한 사람이 상황별로 모드를 바꾸는** 그림이에요.

| 모드 | 커맨드 | 이 방식으로 일할 때 |
|---|---|---|
| **대리응답** | `/answer` | 코치이(내담자) 질문에 "나리님이라면 할 답"을 한다 |
| **토론** | `/spar` | 나리님의 관점에 반론을 던져 논리를 점검한다 |
| **리서치** | `/research` | 커리어 주제를 나리님 프레임으로 조사해 브리핑한다 |

세 모드 모두 **같은 코치 페르소나**를 공유합니다. 옷을 갈아입는 것뿐이지, 사람이 바뀌는 게 아니에요. 그래서 어느 모드로 부르든 나리님의 말투와 프레임이 일관되게 나옵니다.

## [개념] 서브에이전트 — 잠깐 부르는 임시 지원 인력

서브에이전트는 **특정 작업 하나만 맡기고 결과만 받는** 임시 일꾼이에요. 세 가지 특징이 있습니다.

- **임시성** — 필요할 때 생겼다가, 작업이 끝나면 사라집니다. 상주하지 않아요.
- **맥락(컨텍스트) 보호** — 본체 에이전트의 대화 맥락을 어지럽히지 않습니다. 서브에이전트는 자기 몫의 일만 깨끗한 공간에서 처리해요.
- **객관성** — 본체와 분리돼 있어서, 본체가 만든 결과를 **제3자의 눈**으로 볼 수 있습니다.

::: tip 우리 코치의 서브에이전트 = "근거 검수자"
코치가 대리응답(`/answer`)으로 답을 만들면, 그 답이 정말 `knowledge/`에 근거하는지 **따로 검증하는 임시 인력**을 붙일 수 있어요. 이 검수자를 **근거 검수자(Evaluator)** 라고 부릅니다.

핵심은 **분리**예요. 답을 만든 코치가 자기 답을 스스로 채점하면 후해지기 쉽죠. 그래서 답변 생성(Generator)과 근거 검수(Evaluator)를 **서로 다른 일꾼**으로 나눕니다. 검수자는 답이 어떻게 나왔는지에 물들지 않은 채, 출처가 실제로 자료에 있는지만 냉정하게 봐요. 이게 맥락 보호가 주는 객관성입니다.

실제로 이 검수자를 코치에 붙이는 실습은 **2-4(나만의 코치 만들기) 마지막 단계**에서 합니다. 지금은 "이런 역할을 서브에이전트로 둘 수 있다"만 잡으면 충분해요.
:::

## [개념] 오케스트레이션 — 팀장의 배치·조율

에이전트가 여럿이거나, 같은 작업을 여러 번 굴려야 할 때 **어떻게 나눠 붙일지**를 정하는 게 오케스트레이션이에요. 대표적인 5패턴을 훑어봅니다.

| 패턴 | 한 줄 뜻 | 커리어 코치라면 |
|---|---|---|
| **Fan Out (팬아웃)** | 한 작업을 여러 갈래로 펼쳐 동시에 처리 | 한 코치이의 이력서를 "강점·약점·시장성" 갈래로 나눠 동시에 분석 |
| **Specialized Roles (전문 역할 분담)** | 역할별 전담 에이전트로 나눔 | 답변 생성은 코치, 근거 검증은 검수자로 역할을 쪼갬 |
| **Parallel Research (병렬 리서치)** | 여러 주제를 동시에 조사 | 직무 전환 경로·연봉 벤치마크·업계 동향을 한꺼번에 병렬 조사 |
| **Batch Generation (배치 생성)** | 같은 작업을 여러 입력에 일괄 적용 | 여러 코치이의 질문을 한꺼번에 대리응답 초안으로 생성 |
| **Validation Pipeline (검증 파이프라인)** | 생성 → 검증을 파이프라인으로 연결 | 코치가 답 생성 → 검수자가 근거 확인 → 통과분만 통과 |

::: details 커리어 코치엔 오케스트레이션이 좀 무거워요 — 왜 알아둘까요?
1:1 코칭은 대부분 **코치 한 명(에이전트 하나) + 검수자 하나** 로 충분합니다. 여러 에이전트를 동시에 돌리는 무거운 오케스트레이션은 지금 단계엔 필요 없어요.

그래도 이 패턴들을 소개하는 이유는 **확장의 지도**이기 때문이에요. 나중에 코치이가 늘어 "질문 20개를 한 번에"(Batch Generation) 처리하거나, 리서치를 "여러 주제 동시에"(Parallel Research) 돌리고 싶어질 때, 오늘 잡아둔 이름표가 방향을 알려줍니다. 우리가 2-4에서 붙이는 **근거 검수자**도 사실 Validation Pipeline의 가장 단순한 형태예요.
:::

## [개념] 우리 코치를 이 그림에 얹어보면

정리하면 이렇게 서요.

- **에이전트** = 커리어 코치 본체. 3모드(`/answer`·`/spar`·`/research`)로 일하는 **한 명의 정규 팀원**.
- **서브에이전트** = 근거 검수자(Evaluator). 답의 출처를 따로 검증하는 **임시 지원 인력** (2-4에서 실제로 붙임).
- **오케스트레이션** = 코치 + 검수자를 잇는 아주 가벼운 파이프라인. 나머지 패턴은 미래 확장용 지도.

::: tip [체크포인트] 여기까지 잡으셨다면
- 에이전트 = **AI + 도구 + 판단력**을 가진 정규 팀원 (= 우리 코치)
- 서브에이전트 = 맥락을 보호하며 잠깐 일하는 임시 인력 (= 근거 검수자)
- 오케스트레이션 = 여러 일꾼을 배치·조율하는 방식 (5패턴은 확장의 지도)

지금 5패턴을 다 외우실 필요는 없어요 — "코치 한 명 + 검수자 한 명"만 머리에 남으면 충분합니다.
:::
